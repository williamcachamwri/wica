<p>Every FreeRTOS project I've worked on has the same problem.</p>

<p>You have a sensor task that reads temperature. You have a UI task that needs to display it. So you create a QueueHandle_t, pass it to both tasks at init, call xQueueSend on one side and<br>
xQueueReceive on the other. Fine.</p>

<p>Then you add a WiFi task that also needs temperature. You add another queue. Then a logging task. Another queue. Soon your app_init() is a mess of queue handles being passed around, and<br>
changing one task means touching everything it's connected to.</p>

<p>On bare metal it's the same story in a different shape — a dozen global flags in main: if (flag_sensor) ... if (flag_button) ... if (flag_timer) ..., each one added as the project grows,<br>
none of them easy to trace back to where they're set.</p>

<p>On embedded Linux it's pointers — modules holding direct references to each other, so a change in one ripples everywhere.</p>

<p>I wanted one solution that works across all three without rewriting the dispatch logic every time.</p>

<p>embedmq collapses it to 3 functions:</p>

<p>embedmq_register(q, "sensor.temp", on_temp, NULL); // subscriber<br>
embedmq_post(q, "sensor.temp", &amp;data, sizeof(data)); // producer, any thread/task</p>

<p>No shared queue handles. No global flags. No direct pointers between modules. The library handles the ring buffer, the mutex, and the semaphore wakeup.</p>

<p>Same API, three platforms:</p>

<ul>
<li>Linux: pthread + POSIX semaphore, zero external dependencies</li>
<li>FreeRTOS: counting semaphore + xTaskCreate, static mode for zero heap after init</li>
<li>Bare-metal: C11 atomic spinlock, drive dispatch with embedmq_poll() from your superloop</li>
</ul>

<p>FreeRTOS PAL is verified on the POSIX simulator in CI — not real hardware yet, I'll be honest about that.</p>
